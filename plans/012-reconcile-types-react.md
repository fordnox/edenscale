## Why this matters

`pnpm turbo run typecheck` **fails** at the planning commit. It is not caused by
any recent change — it is a pre-existing lockfile inconsistency, and it was
discovered while executing plan 001.

Plan 001 adds a CI workflow whose frontend job runs `pnpm turbo run typecheck`.
That job will fail on arrival until this is fixed. Since the entire point of
plan 001 is to establish a trustworthy verification baseline, shipping a CI job
that is red for an unrelated reason defeats it. **This should land with or
before plan 001's CI workflow.**

The failure also blocks the three SPAs from being typechecked at all: turbo
halts on the upstream `@edenscale/ui` dependency, so `manager`, `investor`, and
`superadmin` typecheck tasks never run. Whatever type errors those apps may
have are currently invisible.

## Current state

`pnpm turbo run typecheck` produces:

```
@edenscale/ui:typecheck: src/components/ui/calendar.tsx(132,15): error TS2322:
  Type 'import(".../@types+react@19.2.14/node_modules/@types/react/index").Ref<HTMLDivElement>'
  is not assignable to type 'React.Ref<HTMLDivElement>'.
@edenscale/ui:typecheck: src/components/ui/spinner.tsx(7,6): error TS2322: ...
  is not assignable to type 'RefAttributes<SVGSVGElement>'.
 Tasks:    3 successful, 4 total
 Failed:    @edenscale/ui#typecheck
```

Two copies of the types package are installed:

```
node_modules/.pnpm/@types+react@19.2.17
node_modules/.pnpm/@types+react@19.2.14
```

The types are structurally identical but nominally distinct, so `Ref<T>` from
one is not assignable to `Ref<T>` from the other.

**The source of the second copy** — `pnpm-lock.yaml:75-80`, in the `apps/emails`
entry, pins an **exact** version while every other package uses a caret:

```yaml
      '@types/react':
        specifier: 19.2.14
        version: 19.2.14
      '@types/react-dom':
        specifier: 19.2.3
        version: 19.2.3(@types/react@19.2.14)
```

Compare `packages/ui/package.json:66-67` and `apps/manager/package.json:37-38`,
which both declare `"@types/react": "^19.2.17"`.

And the root `package.json` already tries to force a single version:

```json
    "overrides": {
      "@types/react": "19.2.17",
      "@types/react-dom": "19.2.3"
    }
```

So the override exists but the lockfile does not reflect it — the lockfile is
stale relative to the override (it was most likely added without a full
reinstall; `package.json` and `pnpm-lock.yaml` were both touched by the recent
`update workspace` / `web manifests` commits).

## Maintenance notes

- **CORRECTION to this plan's diagnosis, found during execution and verified.**
  This plan claimed the root `pnpm.overrides` block failed because the lockfile
  was not regenerated. That is wrong. The override **has never applied at all**:
  pnpm 11.15.1 emits, on every command,
  `The "pnpm" field in package.json is no longer read by pnpm. The following keys were ignored: "pnpm.overrides"`.
  Overrides moved to `pnpm-workspace.yaml`, and this repo's
  `pnpm-workspace.yaml` has no `overrides` section — so the block in the root
  `package.json` is dead configuration.
- **Follow-up required to make this fix durable** (do this when landing the
  plan, it is ~4 lines): move the overrides into `pnpm-workspace.yaml`:
  ```yaml
  overrides:
    "@types/react": "19.2.17"
    "@types/react-dom": "19.2.3"
  ```
  and delete the now-inert `"pnpm": { "overrides": ... }` block from the root
  `package.json`. Without this, step 2's specifier alignment is the *only* thing
  holding the workspace to one `@types/react` — any package that pins a
  different version reintroduces the exact failure this plan fixed, silently.

  pre-existing `node_modules`. But it *will* faithfully reproduce a bad lockfile,
  so the lockfile must be correct in the commit.
