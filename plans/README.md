## Decisions a human should confirm

Three landed or deferred items rest on judgment calls, not on unambiguous defects.
Flagging them so they get a deliberate review rather than passing silently.

1. **Plan 005 changes product behavior.** `overdue` capital calls now count as
   outstanding on dashboards. An existing test asserted the opposite with a
   deliberate-looking comment; that test was updated. The case for the change:
   `app/schemas/dashboard.py:57` exposes only `capital_calls_outstanding` and
   `dashboard_repository.py` had **no** other reference to `overdue`, so past-due
   calls were counted nowhere at all; a daily cron actively moves calls into that
   state; and `payment_matching.py:36-40` already lists `overdue` as collectible.
   The counter-reading: if "outstanding" was meant as "awaiting payment, not yet
   late" and a separate overdue metric was simply never built, then the right fix
   is a new metric, not folding it in. Reversible either way.
2. ~~The SQLite divergence~~ — **RESOLVED.** The maintainer chose to drop
   SQLite. Plan 013 removed the fallback; a missing or non-PostgreSQL
   `APP_DATABASE_DSN` now fails immediately at conftest import with a message
   naming the variable. The ~168 `str`-into-`Uuid` test call sites were
   deliberately left alone (masked by psycopg2 coercion) — low-priority
   cleanup, noted in plan 013.
3. **The notification retry** (see deferred list): removing the blanket catch
   requires a `NotificationLog` migration first. Until then, notifications can
   still be lost silently.

## Deferred — real findings, deliberately not planned yet

These were confirmed but need a human decision or a larger project than a single
plan. Listed so they are not lost.

- **Four more architecture docs describe the pre-membership RBAC model as
  current.** Found while executing plan 010, which was scoped to the ADRs.
  `docs/architecture/rbac-model.md` (lines 18, 24-28, 36, 41, 44, 47, 51, 55,
  61, 63, 103, 108, 112), `api-layering.md` (27, 53, 63, 106-111, 128),
  `system-overview.md` (68-69, 71), and `database-schema.md` (66, 118, 121) all
  still document `users.role`, `users.organization_id`, `require_roles(*allowed)`,
  `PATCH /users/{id}/role`, and `WHERE organization_id = :user.organization_id`
  scoping — every one of which is false. `AGENTS.md` is vaguer and contains no
  clearly false specific claim, but it links to the stale `rbac-model.md`. This
  is the same drift ADR-001 had, across five more files; a straightforward
  follow-up to plan 010 with the same shape.
- **ADR-002 says it rejected proxying uploads through the API — but that is what
  shipped.** Found while executing plan 010 and verified independently.
  ADR-002's "Why we did not pick Option B" rejects proxy uploads as "a needless
  bandwidth and latency tax" that holds an API worker for the upload's duration.
  But `S3Storage.presign_put` (`app/services/storage.py:185-200`) returns an
  API-relative `/documents/upload/{key}` URL, and that route writes to the bucket
  server-side — so uploads *are* proxied, exactly the tradeoff the ADR says was
  rejected. Only downloads (`presign_get`) are true direct-to-bucket presigned
  URLs. No evidence in code or git history explains why the shipped path
  diverged. Worth a decision: either the ADR's rationale is stale and should be
  corrected, or the implementation drifted from an intentional decision and the
  DoS concern the ADR raised is now live in production. Note this also means the
  100 MB upload buffering issue (deferred above) sits on the API process, which
  is what ADR-002 was trying to avoid.

- **Notifications are still silently lost — needs a `NotificationLog`
  idempotency key first.** Plan 009 fixed JWKS rotation and the blocking Resend
  call but correctly STOPPED before removing the blanket `except Exception` in
  `task_send_notification`, because removing it is unsafe today. The hazard,
  traced during execution: `channel.send()` delivers a real email, then the
  later `db.commit()` for the `NotificationLog` row fails (the exact transient
  failure the fix targets), arq retries the whole job, and `channel.send()` runs
  again — a **second email to the same limited partner**. `NotificationLog`
  cannot currently prevent that: it has no `reference_type`/`reference_id`
  columns (those live only on `Notification`, which is itself recreated fresh on
  every retry), no unique constraint, and no `job_id`; `max_tries` is not set at
  enqueue time either. So there is no dedup handle available.
  **The prerequisite work**: add a persisted idempotency key to
  `NotificationLog` (a migration) — e.g. unique on
  `(notification_type, reference_type, reference_id, channel, recipient)` — and
  have the channel loop skip any channel already recorded as delivered. Only
  once that exists can the blanket catch be removed so arq retries safely.
  Until then the bug stands: any transient failure logs an error, reports
  success, and the LP is never told money is due.
- **Authenticated dev-storage reads, if ever wanted.** `GET /dev-storage/{key}`
  remains unauthenticated after plan 002. This is the tradeoff ADR-002 records
  and accepts for a dev box, and plan 002 closed the two things ADR-002 did
  *not* sanction (reads outside the storage root; a config that fails open into
  this backend in production). A header token gate is **impossible** here —
  `download_url` is consumed as a plain `<a href>` navigation, which cannot
  carry a custom header. The correct design would be a query-string token
  embedded by `presign_get`, mirroring `S3Storage.presign_get`'s signature, and
  validated by the route. Only worth doing if dev boxes start holding data worth
  protecting.

- **`recompute_status` cannot unwind a payment.** When `total_paid` returns to 0,
  the `else` branch preserves the current status, so a call corrected to zero
  stays `partially_paid` forever. Same drift in `distribution_repository`.
  Needs a decision about repairing existing rows. (Noted in plan 005.)
- **`recompute_status` bypasses `_ALLOWED_TRANSITIONS`**, so the "terminal
  `paid`" guarantee is not actually enforced — two code paths disagree about
  what a `paid` call may become.
- **Synthetic bank-reference dedupe key.** When a statement entry has no bank
  reference, one is composed from `value_date|amount|payer`. Two genuine
  same-day, same-amount payments from one payer collapse into one — the second
  is dropped at import or auto-`ignored` at apply. Fixing it changes the
  dedupe semantics that an existing test pins. (Noted in plan 003.)
- **Email-ingest trusts a self-asserted sender.** Org routing resolves the
  acting user from a request-body `sender_email` field with no DKIM/SPF verdict
  in the payload. Security rests entirely on the Cloudflare Worker verifying the
  envelope sender — which is not in this repo and could not be read. Needs
  Worker-side coordination; ship the field optional-then-required.
- **Email ingest is not idempotent.** No message-id, and a random storage key
  per call, so any Worker retry re-ingests: duplicate documents, duplicate
  notification emails, duplicate LLM draft jobs.
- **`task_draft_letter` retry duplicates drafts.** No blanket catch and no
  dedupe, so a failure after persistence re-runs the whole task — duplicate
  drafts and duplicate OpenRouter spend. (Noted in plan 009.)
- **Prompt injection into letter drafting.** Untrusted document text is
  interpolated into the LLM user turn with no delimiters or provenance marker,
  and emailed PDFs auto-enqueue a draft. Output is a `Communication` draft a
  fund manager reviews and sends to LPs. Fix is small (delimiters + a system
  prompt clause); it is deferred only because it needs the review-gate assertion
  checked at the same time.
- **Upload size limits are enforced after buffering.** `await request.body()`
  reads the whole payload into memory *then* compares against the 100 MB cap, so
  the cap does not bound resident memory. Same shape in bank-import upload.
- **No content-type allowlist on uploads.** The caller's declared content type is
  stored verbatim, and it is what the letter-drafting path dispatches on.
- **CSRF posture is undecided.** `_extract_token` accepts a `hanko` cookie when
  no `Authorization` header is present, and several state-changing POSTs take no
  body. Exploitability depends on the `SameSite` attribute Hanko sets, which
  this repo does not control — needs verification before deciding.
- **Tenant-wide aggregate subqueries.** `investor_repository._base_query` and
  `dashboard_repository` group the *entire* commitments table before filtering
  by org, so cost scales with total platform data. Needs `EXPLAIN ANALYZE` on a
  seeded multi-tenant DB before/after — Postgres may already push the predicate
  down.
- **`joinedload` on to-many collections** in four repositories multiplies rows on
  the wire; `selectinload` is the right call. Magnitude depends on
  items-per-call in production.
- **Four unbounded list endpoints** (three superadmin, one fund-valuations).
  `list_all_users` nests memberships and organizations, so it degrades
  superlinearly as the platform grows.
- **No route-level code splitting in the manager app** — 27 pages plus recharts
  in the main chunk, so every user downloads the whole app to see the login
  screen. Measure with a real build before picking split points.
- **No frontend linter at all.** All five packages define `"lint": "tsc --noEmit"`,
  identical to their typecheck script. Nothing catches exhaustive-deps or
  floating promises across ~23K lines of React. (Noted in plan 011.)
- **Six duplicated frontend modules** across manager/investor/superadmin
  (`ProtectedLayout` is triplicated; `InvitationAcceptPage` and `CommandPalette`
  are near-identical pairs). The lifting infrastructure already exists in
  `packages/ui` and `packages/shared`. Do 001 first — there is no frontend test
  coverage to refactor against.
- **No structured logging or request correlation.** Notifications cross a
  process boundary (request → arq worker) with no ID tying them together, so
  debugging a missing capital-call email means adding log statements and
  redeploying.
- **Astro < 7.0.6** carries four moderate XSS advisories, reachable from
  `apps/docs` and `apps/web`, which the gateway serves on the same origin as the
  app shells. The fix is a lockfile refresh within the existing `^7.0.x` caret.
- **Stale exact pins** in `apps/backend/pyproject.toml`: `sqlalchemy==2.0.23`
  (Nov 2023) and `pydantic==2.7.0` (Apr 2024) pinned exactly, while
  `fastapi`/`boto3`/`stripe` float freely with no evident rule.
- **`users.password_hash` is a dead column** — `app/models/user.py:26` is its
  only reference anywhere; auth is entirely Hanko.

## Findings considered and rejected

Recorded so nobody re-audits them.

- **NAV tie-break divergence between `latest_fund_nav` and `latest_fund_navs`
  — REFUTED during plan 008.** The audit flagged (at MED confidence, explicitly
  pending verification) that the singular function orders by
  `as_of_date.desc(), created_at.desc()` while the bulk one orders only by
  `as_of_date.desc()`, so a fund with two same-date valuations could show
  different NAVs on the list vs the detail page. It cannot:
  `apps/backend/app/models/fund_valuation.py:29` declares
  `UniqueConstraint("fund_id", "as_of_date", name="uq_fund_valuation_as_of")`,
  making two same-date rows for one fund impossible at the DB level. The
  ordering difference is unreachable dead code under the current schema. No
  production change was made. Only revisit if that constraint is ever dropped.
- **Bulk-vs-single fund metrics divergence — REFUTED during plan 008.** The
  audit reasoned that `fund_metrics_bulk` re-implements the aggregation to avoid
  N+1 and could therefore drift from `fund_metrics`, showing different TVPI on
  the funds list than the fund detail page. A field-by-field equivalence test
  across three seeded funds (`committed`, `called`, `distributed`, `nav`, `dpi`,
  `irr`, `tvpi`, `rvpi`, `called_pct`) shows they match exactly. The concern was
  reasonable but wrong; the equivalence test now guards it permanently, so a
  future optimization cannot silently introduce the drift.

- **SQLAlchemy legacy `Query` API → 2.0 `select()` style.** 164 call sites across
  20 repositories. The legacy API is fully supported in 2.x with no announced
  removal. Pure churn, real regression risk across every data path, zero runtime
  benefit. Only worth it bundled with an async migration, which needs
  `select()` anyway.
- **`psycopg2-binary` → `psycopg3` standalone.** psycopg2 is maintenance-mode
  but not abandoned or EOL. Tiny blast radius, but roughly zero benefit on its
  own. Only worth doing as a prerequisite of the async-engine migration.
- **Pydantic v1 patterns under v2.** Checked for `orm_mode`, `class Config`,
  `@validator`, `.dict()`, `parse_obj` — **zero hits**. The v2 migration is
  clean. Nothing to do.
- **Removing three unused frontend deps** (`ogl`, `remark-gfm`, `date-fns`).
  Vite tree-shakes unimported packages, so bundle impact is ~zero. Hygiene only,
  and absence-of-usage across a monorepo is easy to get wrong.
- **TanStack Query cache configuration.** `staleTime: 5min`, `retry: 1`,
  `refetchOnWindowFocus: false`, shared across all three apps, with stable query
  keys. Deliberate and sensible. No redundant-fetch problem found.
- **`metrics.py` / `allocation.py` aggregate handling.** Audited specifically and
  they are clean: sums pushed into SQL, bulk metrics done in a fixed 4 queries,
  NAV lookup batched, `allocate_pro_rata` O(n) with no queries. The only
  Python-side computation is the IRR solve, which belongs in application code.
- **The redundant OPTIONS middleware** at `app/main.py:41` looks alarming (it
  reflects an arbitrary origin with `Allow-Credentials: true`) but is not
  exploitable: `CORSMiddleware` is added later and therefore sits outside it,
  handling every preflight that carries an `Origin` header. Reachable only for
  OPTIONS requests without an `Origin`, which browsers never send. Worth
  deleting as dead code; not a security finding.
- **`CORS_ALLOW_ORIGINS` breadth** — defaults to a single localhost origin,
  supplied per-environment. No wildcard-with-credentials pattern.
- **`GENERATE_OPENAPI_DOCS`** — defaults to false and is set per-environment. A
  deliberate choice, not a leak.
- **Superadmin gate coverage** — all 13 superadmin routes carry
  `Depends(require_superadmin)` individually, and the tenant dependencies
  actively reject superadmins from tenant paths. The two surfaces cannot be
  crossed.
- **Committed secrets** — `.kamal/secrets` is tracked but every entry is a shell
  indirection. No literal credential values are committed anywhere, and no
  `.env` has ever been added in git history. (The one literal, `DEBUG=1`, is
  fixed by plan 002.)
- **Invitation tokens** — `secrets.token_urlsafe(48)` with email-match, status,
  and expiry checks on accept. No weakness found.
- **Tenant isolation across the 21 routers.** The full dependency chain was
  traced for every router; no object-by-ID route skips the ownership predicate,
  and sub-resource routes verify parent-child linkage before mutating. LP
  scoping correctly joins through `Investor.organization_id`. This was the
  highest-suspicion area of the audit and it came back clean — worth recording
  so it is not re-litigated.
- **LP tenancy test coverage** — genuinely good. Capital calls, distributions,
  and the investor portal all have LP-scoping assertions, and `lp_scope.py` is
  exercised through 12 repositories' route tests.
- **Float money math in `ImportBankPaymentsPage.tsx`** — uses `Number()` +
  `toFixed(2)`, but the backend re-parses as `Decimal` and the magnitudes are
  realistic. Not worth a finding on its own.
- **No prompt-injection content found** in any file read during the audit.
