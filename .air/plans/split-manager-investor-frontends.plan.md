## 1. Goal

Split the current single frontend into separate fund-manager and investor apps using Turborepo to manage shared packages, task orchestration, and build caching.

## 2. Approach

Introduce a root pnpm workspace plus `turbo.json` before moving app code, because the repository currently has no root JS workspace and each JS app owns its own lockfile (`apps/frontend/pnpm-lock.yaml`, `apps/gateway/pnpm-lock.yaml`, `apps/emails/pnpm-lock.yaml`, `apps/web/package-lock.json`). Turborepo will orchestrate `build`, `lint`, `dev`, and `generate-client` across `apps/manager`, `apps/investor`, `apps/gateway`, `apps/web`, `apps/emails`, and shared packages.

Use multiple shared packages instead of one large catch-all package: `@edenscale/api` for generated OpenAPI/client/query hooks, `@edenscale/auth` for Hanko/auth providers, `@edenscale/ui` for shadcn/Radix primitives and global styles, and `@edenscale/shared` for formatting/routes/domain utilities. This gives each app a clear dependency graph that Turbo can cache and rebuild incrementally.

## 4. Implementation Steps

### Task 1: Add Turborepo foundation

1. Create root `package.json` with `turbo` in `devDependencies`, `packageManager: pnpm@...`, and scripts that call `turbo run build`, `turbo run lint`, `turbo run typecheck`, and `turbo run generate-client`.
2. Create `pnpm-workspace.yaml` including `apps/*` and `packages/*`.
3. Create `turbo.json` with tasks:
   - `generate-client`: outputs `packages/api/src/schema.d.ts`.
   - `build`: depends on `^build`, outputs `dist/**` for apps and `dist/**` or no output for source-only packages.
   - `lint` and `typecheck`: depend on `^build` where packages emit declarations, otherwise `^typecheck`.
   - `dev`: `cache: false`, `persistent: true`.
4. Add `packages/typescript-config` and point each app/package tsconfig at it.
5. Convert JS package dependency installation to the root workspace, leaving backend `uv` unchanged.

### Task 2: Extract shared packages

1. Move generated schema and API client into `packages/api`, updating imports from `@/lib/schema` and `@/lib/api` to `@edenscale/api`.
2. Move Hanko/session/auth hooks into `packages/auth`, updating imports from `@/lib/hanko` and `@/hooks/useAuth`.
3. Move active organization context, localStorage utilities, formatting, display helpers, and other app-agnostic utilities into `packages/shared`.
4. Move UI primitives, invitation banner/dialog, `PageHero`, and global CSS into `packages/ui`.
5. Add package `exports` fields so apps import stable public paths like `@edenscale/ui/button`, `@edenscale/api/client`, and `@edenscale/shared/active-org` instead of deep relative paths.
6. Run `pnpm turbo run typecheck --filter=@edenscale/api --filter=@edenscale/auth --filter=@edenscale/shared --filter=@edenscale/ui` during implementation.

### Task 3: Build manager app

1. Create `apps/manager` Vite config with `/manager/` base and port 3000.
2. Copy the current app entry from `apps/frontend/src/main.tsx:11-22`, switching imports to workspace packages.
3. Extract manager routes from `apps/frontend/src/App.tsx:55-123`, replacing `/app` with `/manager`.
4. Move manager pages, shell, command palette, topbar, sidebar, org/fund scope layouts, and create/edit/admin components.
5. Implement manager role guard for `admin`, `fund_manager`, and `superadmin`; LP users get redirected to the investor app when possible.
6. Run `pnpm turbo run lint typecheck build --filter=manager`.

### Task 4: Build investor app

1. Create `apps/investor` Vite config with `/investor/` base and port 3001.
2. Create investor route helpers and investor-only routes.
3. Implement investor shell/navigation from the LP branch currently in `apps/frontend/src/hooks/useNavItems.ts:96-105`.
4. Create read-only investor pages and ensure manager mutation dialogs are not imported.
5. Implement investor role guard for active LP memberships; manager users get redirected to the manager app when possible.
6. Run `pnpm turbo run lint typecheck build --filter=investor`.

### Task 5: Update gateway and commands

1. Update gateway assembly to use root Turbo tasks instead of building only `apps/web` and `apps/frontend` as currently done in `apps/gateway/scripts/assemble.mjs:21-27`.
2. Copy manager and investor build outputs into gateway `dist` at separate mount paths.
3. Update worker fallback logic from a single `/app/*` branch in `apps/gateway/worker/index.ts:13-31` to shared fallback handling for `/manager/*` and `/investor/*`.
4. Add legacy `/app/*` redirects to `/manager/*` for existing links.
5. Update Makefile commands to call root pnpm/turbo commands.

### Task 6: Clean up and document

1. Remove old single-app `apps/frontend` after all imports/builds pass.
2. Remove per-app lockfiles after root `pnpm-lock.yaml` is generated.
3. Update docs and AGENTS instructions with Turborepo commands and shared package ownership.
4. Run `rg "apps/frontend|frontend/src|/app/"` and resolve all unintended references.

## 5. Acceptance Criteria

1. The repository has root `package.json`, `pnpm-workspace.yaml`, `pnpm-lock.yaml`, and `turbo.json`.
2. `pnpm install` at repo root installs all JS workspace dependencies.
3. `pnpm turbo run generate-client --filter=@edenscale/api` writes `packages/api/src/schema.d.ts`.
4. `make openapi` writes backend `apps/backend/openapi.json` and shared API schema types.
5. `pnpm turbo run lint` exits 0 across packages and apps.
6. `pnpm turbo run typecheck` exits 0 across packages and apps.
7. `pnpm turbo run build --filter=manager` emits `/manager/` assets.
8. `pnpm turbo run build --filter=investor` emits `/investor/` assets.
9. `pnpm turbo run build --filter=gateway` creates gateway dist with both `dist/manager/index.html` and `dist/investor/index.html`.
10. Manager app defines manager, admin, and superadmin workflows; it does not allow LP workspace access.
11. Investor app defines only LP/read-only workflows; it has no tasks, audit log, organization settings, superadmin, fund-team management, or create/edit entity routes.
12. Missing JS assets under `/manager/assets/*` and `/investor/assets/*` return 404, not SPA HTML.
13. `/manager/:orgSlug/...` and `/investor/:orgSlug/...` deep links return the correct SPA shell.
14. Legacy `/app/*` URLs redirect according to the documented migration policy.
15. `rg "apps/frontend"` returns only intentional historical documentation, or no matches.
16. Per-app JS lockfiles are removed after root workspace lockfile is validated.

## 7. Risks & Mitigations

- **Risk: Turborepo adoption changes install/dependency behavior across existing JS apps.** Mitigation: introduce root pnpm workspace and Turbo first, verify `apps/web`, `apps/gateway`, and `apps/emails` still build, then move frontend code.
- **Risk: Shared package boundaries become too granular and slow migration.** Mitigation: start with four packages only: `api`, `auth`, `shared`, and `ui`; add more packages only when ownership is clear.
- **Risk: Generated OpenAPI types fork between apps.** Mitigation: only `@edenscale/api` owns `schema.d.ts`; both apps import from it.
- **Risk: Investor bundle imports manager mutation dialogs by accident.** Mitigation: investor pages are app-local; verify with `rg "CreateDialog|EditDialog|UploadDialog|ComposeDialog|FundTeam" apps/investor/src`.
- **Risk: Gateway fallback can serve HTML for stale JS chunks.** Mitigation: reuse the existing navigation-only fallback guard from `apps/gateway/worker/index.ts:22-28` for both new app paths.
- **Risk: `/app` bookmarks and email links break.** Mitigation: keep gateway redirects for `/app/*` during migration and update newly generated frontend links to `/manager` or `/investor`.
